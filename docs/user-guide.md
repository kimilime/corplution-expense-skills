# Corplution Reimbursement Wizard 使用指南

Corplution Reimbursement Wizard 用于把发票、行程单、付款凭证、审批截图和项目说明整理成可提交的恺讯报销申请表及文件包。正常使用时，你不需要操作脚本、填写 JSON 或自己套 Excel 模板，只需上传材料、说明项目情况，并在聊天中确认少量不确定事实。

## 它能做什么

- 读取带文本层的 PDF、扫描件和图片；需要 OCR 时自动检查环境，无法可靠识别时明确要求复核，不猜测内容。
- 提取发票号码、发票类型、开票单位、日期、金额、费用类别、路线、住宿日期等信息，并保留原始文件名供你核对。
- 识别并关联滴滴/高德行程单、付款凭证、替票审批截图等支持材料；一张汇总打车发票可拆成多笔实际行程填写。
- 按高铁/飞机、酒店、打车、餐费、通讯费和其他费用分别判断，不使用一条简单的“城市匹配”规则处理所有费用。
- 根据项目日期、城市、客户、Client Charge Code、路线和你的自然语言记录进行预归集，并把真正不确定的项目按类别合并询问。
- 逐条核对你提供的费用记录是否找到对应凭证，避免“记录里写了，但流程静默漏掉”。
- 检查重复材料、缺票、缺行程单、日期不明确、项目归属冲突、替票审批、餐费和酒店标准等问题。
- 在宿主支持独立子 Agent 时，写表前运行证据复核与财务预审，进一步发现“看起来合理但其实归错”或“财务可能不接受”的费用。
- 直接生成带项目分块、小计、公式、总额和状态校验的 Excel，并整理发票、支持文档和可追溯清单。

## 安装

复制整个 `corplution-reimbursement-wizard` 文件夹，而不只是 `SKILL.md`。其中的脚本、规则和资源文件都需要一起保留。

| 使用环境 | 安装位置 |
| --- | --- |
| Codex | `C:\Users\<你>\.codex\skills\corplution-reimbursement-wizard\` |
| WorkBuddy | `C:\Users\<你>\.workbuddy\skills\corplution-reimbursement-wizard\` |
| Claude Code（个人） | `~/.claude/skills/corplution-reimbursement-wizard/` |
| Claude Code（单一项目） | `<项目根目录>/.claude/skills/corplution-reimbursement-wizard/` |

WorkBuddy 也可通过“专家 > 技能 > 上传技能”直接安装完整 ZIP 压缩包。安装或更新后，请重新打开一个 task/session。

首次运行时，Agent 会自行检查 Python 和 OCR 依赖。普通电子发票通常可直接处理；扫描件或图片缺少必要 OCR 工具时，Agent 会说明需要补装什么或请你人工确认。

## 怎么开始

在 Codex 或 WorkBuddy 中，可以直接提及：

```text
$corplution-reimbursement-wizard
```

在 Claude Code 中，可以输入 `/corplution-reimbursement-wizard`，也可以直接说“帮我按 Corplution 流程处理报销材料”。

随后上传材料并自然描述情况，例如：

```text
请处理我上传的本期发票和行程单。
报销周期是 6 月 1 日至 6 月 30 日，Requester 是张三。
6 月 8 日至 12 日在郑州 A 客户项目，Charge Code 使用当前共享 BD Code；
6 月 18 日至 20 日在北京 B 客户项目，Code 是实际项目编号。
6 月 8 日去虹桥站的打车是郑州项目接驳；
6 月 10 日餐费 133.5 元，和张三一起。
```

你不必提前分类、改名或整理表格。信息可以在过程中继续补充。

## 应提供什么

尽量一次性提供同一报销周期内的材料：

1. **发票**：PDF、图片、扫描件或其他原始电子凭证。
2. **支持材料**：滴滴/高德行程单、付款凭证、审批截图、订单或行程证明。
3. **项目说明**：每个项目的日期范围、城市、客户名称、实际 Client Charge Code，以及必要的出差或接驳说明。
4. **费用记录**：你自己记录的日期、商户、金额和项目，例如“6.1 德克士 61.8”。金额或商户不完全准确也可以，系统会结合日期、金额和文本综合匹配。
5. **特殊事实**：共同用餐人、加班、合住、部分报销、不报销、替票、退票、特殊活动餐标等。

同一个 Code 对应不同 Client 时，仍视为不同项目。共享 BD 或 ADMIN Code 随财年变化时，告诉 Agent 使用当前配置或提供本期实际 Code；不要沿用旧年份示例。

不确定的内容不用勉强补齐。Agent 会先做有依据的推断，再集中询问剩余问题。

## 常见费用会怎样处理

| 费用 | 主要判断依据 | 你可能需要补充 |
| --- | --- | --- |
| 高铁 / 飞机 | 实际出行日期、路线、目的地项目；同日连续换乘先组成完整旅程，再按最终去向或刚结束的项目判断 | 中途是否真正停留、改签或退票情况 |
| 退票费 | 退票凭证上的退票费就是报销金额，项目归属沿用原行程逻辑 | 退票对应哪段行程 |
| 酒店 | 入住/离店日期、晚数、酒店城市和项目行程 | 发票未写住宿日期时补充实际入住信息；合住时说明同住人 |
| 打车 | 行程单中的日期、发生城市、起终点类型，以及前后高铁/航班和项目行程 | 无法判断的地点是家、公司、客户、酒店、餐厅还是其他地点；是否加班 |
| 餐费 | 优先综合你的日期、商户和金额记录；开票日期通常不直接当作就餐日期 | 实际就餐日、项目、共同用餐人、加班或机场/车站场景 |
| 通讯费 | 账期月份，日期记为该月最后一天，使用当前 ADMIN Code，Client 写“通讯费” | 通常无需额外说明 |
| 其他费用 | 发票内容和你说明的实际事项 | 事项名称、项目归属和记账日期；不明确时会提示确认 |

Excel 的费用列和 `Expense Nature` 按实际发生地执行“形式重于实质”：上海发生的餐费进入 `meal`、上海发生的打车进入 `taxi`；外地餐费和打车进入 `travel`。这不会改变费用所属项目。

餐费标准按业务场景判断，而不是按 Excel 列判断：出差餐费通常为每天 150 元；只有明确的本地加班餐费适用每天 60 元。北京、上海、广州、深圳酒店通常为每晚 800 元，其他城市每晚 600 元。共同用餐、合住或经批准的特殊情况会作为说明或提示，不会被系统机械地直接删除金额。

年会、半年会等活动如有一次性餐标，可以直接说明具体日期、标准和活动名称。该标准只对指定活动和日期生效，不会改动公司的通用规则。

替票或抵用发票必须由你明确指出，并提供对应审批截图；最终 Note 会标注“（抵）”。如果实际报销金额低于发票金额，Note 会同时说明发票金额和实际报销金额。

## 过程中会发生什么

通常会有两到几轮对话：

1. **识别材料**：Agent 在聊天中列出编号、原文件名、类型、销售方、日期、金额和识别状态。你可以直接指出“第 9 项不对”或按原文件名更正。
2. **预归集项目**：系统按费用类型和行程做初步匹配，不会把所有不确定费用都丢给你。仍不确定的餐费、打车、酒店等会分类汇总询问。
3. **核对用户记录与材料**：如果你的记录没有找到发票，Agent 会询问它是由某张汇总发票覆盖、稍后补票，还是不报销；不会静默忽略。
4. **补件与修正**：缺少行程单、审批截图、住宿日期或关键事实时，Agent 会说明缺什么、为什么需要。你可以补文件，也可以明确某项不报销。
5. **独立复核**：在环境支持真正独立子 Agent 时，Otako 会重新核对归集与证据、路线和金额；Kaede 会从财务审批角度检查合规性、审批材料、业务理由和是否漏报。它们只读审查，不会自行改账。
6. **写表与标准检查**：确认 Requester 后生成 Excel，并在聊天中展示餐费、酒店及其他提示。需要你决定的问题未解决前，不会把结果称为可提交。
7. **打包**：只有文件和清单校验通过后，才生成最终可提交包。

## 更正、补票和增删材料

你可以始终使用自然语言修改，例如：

- 第 12 项金额和原发票不一致，请先查源文件再修改。
- 这张滴滴发票缺行程单，我稍后补，现在保持待确认。
- 6 月 3 日的餐费不报销。
- 太原到郑州的高铁应归郑州项目，请重新检查前后行程。
- 这张是替票，审批截图就是我刚上传的图片。

如果中途新增或删除发票、行程单或审批截图，系统会从原始证据重新建立当前批次，安全继承仍然有效的确认，只重新询问发生变化的项目。旧批次中消失的费用会明确列出来，要求确认是删除、由新材料替代，还是需要恢复，不会悄悄消失。

完全重复的文件会在识别阶段要求保留一份，避免重复报销；不会通过删除费用项来代替处理源文件。请保留原始材料，直到最终包确认可提交。

不要手工修改过程 JSON、Excel 中间文件或脚本。若流程中断，直接让 Agent“检查当前进度并继续下一步”。

## 最终会得到什么

正常完成后的目录类似：

```text
报销申请表-Requester-YYYYMMDD/
  报销申请表-Requester-YYYYMMDD.xlsx
  发票/
    001-高铁-446.00.pdf
    002-酒店-499.00-专票.pdf
  支持文档/
    001-行程单.pdf
    002-合伙人审批.png
  package-manifest.md
  package-manifest.json
```

- Excel 按 `Client + Client Charge Code` 分块，项目内部按高铁/飞机、酒店、打车、高德、餐费、通讯费和其他费用排序。
- 每个项目带小计，底部包含 Total、Grand Total 和 `TRUE/FALSE` 状态校验。
- 同一张滴滴/高德汇总发票可对应多笔行程，Excel 中逐笔填写，但共享同一发票编号。
- 发票按编号、类型、金额和专票标识重新命名；支持文档与对应发票编号关联。原始文件不会被移动或改名。
- `package-manifest.md/json` 记录文件数量、关联关系、哈希和阻断问题，便于复核与追溯。

如果 Agent 明确说这是“待补件核对包”或仍有 blocking issue，该目录不能提交。补齐发票、行程单、审批或确认后，应重新生成最终包。

## 使用时请注意

- 发票开具日期不一定是实际消费日期。除通讯费和少数可临时使用发票日期的其他费用外，请优先提供真实发生日期。
- 用户记录可以不完全准确，但请尽量保留日期、金额、商户和项目线索；系统会综合判断并把不确定匹配交给你确认。
- 看到“待确认”“需要补件”或独立复核的 blocking finding 时，先解决对应问题，再提交报销。
- 独立子 Agent 是额外复核，不替代确定性的金额、公式和政策检查；宿主不支持独立子 Agent 时，流程仍可依靠内置检查继续。
- 发票、客户名称、行程和审批截图可能包含敏感信息，只应在公司批准的私有环境中处理和保存。

你负责确认真实事实，Skill 负责识别、推断、核对、写表和打包。信息越完整，追问越少；信息不完整时，它应当询问，而不是编造。
